## 二、MQTT 实践

### 2.1 Mosquitto Broker — 消息转发中心

```
发布者 ──msg──→ [Mosquitto Broker] ──msg──→ 订阅者A
                 端口 1883                    ──msg──→ 订阅者B
                                              ──msg──→ 订阅者C
```

类比：微信群平台。不生产数据，只转发。谁发了消息，自动推给所有进了这个群的人。

| 概念 | 类比 |
|------|------|
| Mosquitto Broker | 微信群这个平台 |
| Topic `factory/lineA/...` | 群名 |
| Publish（发布） | 在群里发消息 |
| Subscribe（订阅） | 进群收消息 |
| `#` 通配符 | 匹配所有子 Topic |
| 端口 1883 | 平台入口 |

### 2.2 MQTT Broker 怎么给多个订阅者发数据

MQTT 用 **Topic（主题）** 做路由，和微信群差不多：

- **发布者** 往某个 Topic 发了消息
- Broker 维护着一张"谁订了什么 Topic"的路由表
- **所有订阅了这个 Topic 的人** 都会立刻收到消息
- `factory/lineA/#` 里的 `#` 是通配符 → 订了 `factory/lineA/` 下面所有子话题

```
发布者: publish("factory/lineA/station01/cylinder/position", "31.0")
                                                        ↓
                                            [Mosquitto Broker]
                                                     │
                          ┌──────────────────────────┼──────────────────────────┐
                          ↓                          ↓                          ↓
                    订阅者A                    订阅者B                    订阅者C
                订了 "factory/lineA/#"    订了 "factory/#"          订了 "#"
                    收到 ✅                   收到 ✅                   收到 ✅
```

**和 OPC UA 订阅的区别**：
| | OPC UA 订阅 | MQTT 订阅 |
|---|---|---|
| 管订阅的人 | OPC UA Server 自己 | Broker（独立的中间人） |
| 谁收消息 | 只有你（点对点） | 所有订了这个 Topic 的人（广播） |
| 用在哪 | 工控机直连 PLC | 跨网络、跨系统、一对多 |

OPC UA 订阅 = 你加了一个 PLC 的微信好友，它定时给你发状态
MQTT 订阅 = 你进了一个群，群里任何人说话你都能听到

### 2.3 OPC UA → MQTT Bridge — 翻译 + 转发

```
opcua_server.py          opcua_mqtt_bridge.py         Mosquitto Broker
(模拟PLC)    ──OPC UA──→  (订阅+翻译+转发)  ──MQTT──→  (1883端口)
  4840端口                  14个NodeId                  ↓
                        NodeId → Topic 翻译      mosquitto_sub / 前端看板
```

Bridge 做的事：
1. 用 OPC UA Client 连接 Server（端口 4840）
2. **订阅** 14 个数据节点的变化（不是轮询！）
3. 收到变化后，把 NodeId 翻译成 MQTT Topic
4. 包装成 JSON `{"ts": "时间戳", "value": 数值, "quality": 质量码}`
5. 发布到 Mosquitto Broker

### 2.4 翻译对照逻辑

OPC UA 说的是 "NodeId `ns=2;i=3` 的值变了"
↓ 经过 Bridge
MQTT 说的是 "向 Topic `factory/lineA/station01/cylinder/position` 发了一条消息"

NodeId → Topic 映射表就是 Bridge 代码里的 `NODE_MAP` 字典。

### 2.5 MQTT 消息格式

**MQTT 不限制消息格式**——可以是 JSON、纯数字、二进制、纯文本，什么都行。Bridge 选 JSON 是因为前端 `JSON.parse()` 一下就转成对象了，好解析。

### 2.6 QoS（服务质量）

| QoS | 含义 | 类比 |
|------|------|------|
| 0 | 发一次，不管收没收到 | 扔纸飞机 |
| 1 | 至少送达一次，丢了重发 | 发顺丰，没签收就重发 |
| 2 | 恰好送达一次，绝不重复 | 签收确认，不能重复签 |

Bridge 用 `qos=1`（确保送达）。

### 2.7 MQTT vs HTTP 轮询

| | HTTP 轮询 | MQTT 订阅 |
|---|---|---|
| 模型 | 你问一次，他给一次（拉） | 有变化自动推给你（推） |
| 带宽 | 每 500ms 问一次，问 100 次可能只有 3 次变化 | 有变化才发，省 97% 带宽 |
| 适用 | Web 网页 | 工厂设备通信 |

## 三、踩过的坑

1. **paho-mqtt 2.x 弃用警告**: `Client()` 必须加 `mqtt.CallbackAPIVersion.VERSION2`
2. **asyncua subscribe_data_change 参数名**: 直接传 `queuesize` 和 `sampling_interval`，不需要 `MonitoringParameters` 对象
3. **winget 安装 Mosquitto**: `winget install EclipseFoundation.Mosquitto`
4. **mosquitto_sub 接收不到消息**: 先启动订阅者再发消息（MQTT 默认不存历史消息）

## 四、命令速查

```bash
# 启动 Mosquitto
net start mosquitto   # 或手动: "C:\Program Files\mosquitto\mosquitto"

# 检查端口
netstat -ano | findstr ":1883"

# 订阅 Topic（# 匹配所有子层级）
"C:\Program Files\mosquitto\mosquitto_sub" -h localhost -t "factory/#" -v

# 手动发布测试
"C:\Program Files\mosquitto\mosquitto_pub" -h localhost -t "test/hello" -m "world"
```
