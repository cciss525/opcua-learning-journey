## 二、MQTT 实践

### 2.1 Mosquitto Broker — 消息转发中心

```
发布者 ──msg──→ [Mosquitto Broker] ──msg──→ 订阅者A
                 端口 1883                    ──msg──→ 订阅者B
                                              ──msg──→ 订阅者C
```

类比：微信群平台。不生产数据，只转发。谁发了消息，自动推给所有进了这个群的人。

| 概念 | 类比 |
|------|------|
| Mosquitto Broker | 微信群这个平台 |
| Topic `factory/lineA/...` | 群名 |
| Publish（发布） | 在群里发消息 |
| Subscribe（订阅） | 进群收消息 |
| `#` 通配符 | 匹配所有子 Topic |
| 端口 1883 | 平台入口 |

### 2.2 MQTT Broker 怎么给多个订阅者发数据

MQTT 用 **Topic（主题）** 做路由，和微信群差不多：

- **发布者** 往某个 Topic 发了消息
- Broker 维护着一张"谁订了什么 Topic"的路由表
- **所有订阅了这个 Topic 的人** 都会立刻收到消息
- `factory/lineA/#` 里的 `#` 是通配符 → 订了 `factory/lineA/` 下面所有子话题

```
发布者: publish("factory/lineA/station01/cylinder/position", "31.0")
                                                        ↓
                                            [Mosquitto Broker]
                                                     │
                          ┌──────────────────────────┼──────────────────────────┐
                          ↓                          ↓                          ↓
                    订阅者A                    订阅者B                    订阅者C
                订了 "factory/lineA/#"    订了 "factory/#"          订了 "#"
                    收到 ✅                   收到 ✅                   收到 ✅
```

**和 OPC UA 订阅的区别**：
| | OPC UA 订阅 | MQTT 订阅 |
|---|---|---|
| 管订阅的人 | OPC UA Server 自己 | Broker（独立的中间人） |
| 谁收消息 | 只有你（点对点） | 所有订了这个 Topic 的人（广播） |
| 用在哪 | 工控机直连 PLC | 跨网络、跨系统、一对多 |

OPC UA 订阅 = 你加了一个 PLC 的微信好友，它定时给你发状态
MQTT 订阅 = 你进了一个群，群里任何人说话你都能听到

### 2.3 OPC UA → MQTT Bridge — 翻译 + 转发

```
opcua_server.py          opcua_mqtt_bridge.py         Mosquitto Broker
(模拟PLC)    ──OPC UA──→  (订阅+翻译+转发)  ──MQTT──→  (1883端口)
  4840端口                  14个NodeId                  ↓
                        NodeId → Topic 翻译      mosquitto_sub / 前端看板
```

Bridge 做的事：
1. 用 OPC UA Client 连接 Server（端口 4840）
2. **订阅** 14 个数据节点的变化（不是轮询！）
3. 收到变化后，把 NodeId 翻译成 MQTT Topic
4. 包装成 JSON `{"ts": "时间戳", "value": 数值, "quality": 质量码}`
5. 发布到 Mosquitto Broker

### 2.4 翻译对照逻辑

OPC UA 说的是 "NodeId `ns=2;i=3` 的值变了"
↓ 经过 Bridge
MQTT 说的是 "向 Topic `factory/lineA/station01/cylinder/position` 发了一条消息"

NodeId → Topic 映射表就是 Bridge 代码里的 `NODE_MAP` 字典。

### 2.5 MQTT 消息格式

**MQTT 不限制消息格式**——可以是 JSON、纯数字、二进制、纯文本，什么都行。Bridge 选 JSON 是因为前端 `JSON.parse()` 一下就转成对象了，好解析。

### 2.6 QoS（服务质量）

| QoS | 含义 | 类比 |
|------|------|------|
| 0 | 发一次，不管收没收到 | 扔纸飞机 |
| 1 | 至少送达一次，丢了重发 | 发顺丰，没签收就重发 |
| 2 | 恰好送达一次，绝不重复 | 签收确认，不能重复签 |

Bridge 用 `qos=1`（确保送达）。

### 2.7 MQTT vs HTTP 轮询

| | HTTP 轮询 | MQTT 订阅 |
|---|---|---|
| 模型 | 你问一次，他给一次（拉） | 有变化自动推给你（推） |
| 带宽 | 每 500ms 问一次，问 100 次可能只有 3 次变化 | 有变化才发，省 97% 带宽 |
| 适用 | Web 网页 | 工厂设备通信 |

## 三、踩过的坑

1. **paho-mqtt 2.x 弃用警告**: `Client()` 必须加 `mqtt.CallbackAPIVersion.VERSION2`
2. **asyncua subscribe_data_change 参数名**: 直接传 `queuesize` 和 `sampling_interval`，不需要 `MonitoringParameters` 对象
3. **winget 安装 Mosquitto**: `winget install EclipseFoundation.Mosquitto`
4. **mosquitto_sub 接收不到消息**: 先启动订阅者再发消息（MQTT 默认不存历史消息）

## 五、配置文件驱动架构

### 5.1 为什么需要配置文件

改造前：NodeId、连接地址、轮询间隔全写死在 Python 代码里，换 PLC 就要改代码。

改造后：一份 `config.yaml` 管所有参数，换 PLC 只改这一个文件，不碰代码。

### 5.2 config.yaml 结构

```yaml
opcua:
  endpoint: opc.tcp://192.168.1.100:4840   # 换 PLC，改这里
  poll_interval: 0.5

database:
  driver: ODBC Driver 17 for SQL Server
  server: localhost\MSSQLSERVER2019
  database: OPCUA
  trusted_connection: "yes"

nodes:           # 采集哪些数据点，新增就加一行
  - id: ns=2;i=3
    group: 工位1·气缸
    name: 位置
    unit: mm
```

### 5.3 代码改造对比

```python
# 改造前 — 写死的
NODES = {
    "ns=2;i=3": ("工位1·气缸", "位置", "mm"),
    ... 13 行 ...
}
client = Client(url="opc.tcp://localhost:4840")

# 改造后 — 读配置
endpoint, poll_interval, NODES = load_config()   # 这一行替掉上面所有
client = Client(url=endpoint)
```

### 5.4 在工厂现场怎么用

```
插上网线连 PLC → 改 config.yaml 里的 endpoint + nodes → python src/collector.py → 数据开始入库
```

## 六、OPC UA 采集引擎

### 6.1 整体架构

```
┌─ config.yaml ──────────────────────────────┐
│  连接信息 + 采集节点 + 数据库配置               │
└──────────────┬────────────────────────────┘
               ↓
┌─ src/collector.py ─────────────────────────┐
│  1. load_config() — 读配置文件                │
│  2. get_db_connection() — 连 SQL Server      │
│  3. init_database() — 建表（不存在则新建）      │
│  4. collect() — 轮询 OPC UA + 写入数据库       │
│     ├ 每 500ms 读 14 个节点                    │
│     ├ 全部失败 → 抛异常 → 外层重连              │
│     └ asyncio.wait_for(0.3s) 防止 TCP 卡死     │
└──────────────┬────────────────────────────┘
               ↓
┌─ SQL Server (OPCUA.opcua_data) ────────────┐
│  id | server_name | node_id | group_name    │
│     | var_name | value | unit | collected_at│
└─────────────────────────────────────────────┘
```

### 6.2 浏览模式：`--browse`

连上 PLC 后自动列出所有变量节点，跳过 Server/Aliases 内部节点：

```
python src/collector.py --browse

[对象] Station01  (ns=2;i=1)
  [对象] Cylinder_01
    [变量] Position  NodeId=ns=2;i=3
    [变量] Speed     NodeId=ns=2;i=4
  [对象] ServoMotor_01
    [变量] ActualRPM  NodeId=ns=2;i=7
    ...
```

用途：到新 PLC → 先浏览 → 把需要的 NodeId 抄到 config.yaml → 开始采集。

### 6.3 数据库表结构

```sql
CREATE TABLE opcua_data (
    id BIGINT IDENTITY PRIMARY KEY,
    server_name NVARCHAR(100),    -- 来源服务器
    node_id NVARCHAR(50),         -- NodeId
    group_name NVARCHAR(100),     -- 分组（工位1·气缸）
    var_name NVARCHAR(100),       -- 变量名（位置）
    value FLOAT,                  -- 数值
    unit NVARCHAR(20),            -- 单位
    quality INT DEFAULT 0,        -- 质量码
    collected_at DATETIME2        -- 采集时间戳
);
```

### 6.4 查询历史数据

```sql
USE OPCUA;

-- 最新 10 条
SELECT TOP 10 * FROM opcua_data ORDER BY collected_at DESC;

-- 某个时间段
SELECT * FROM opcua_data
WHERE collected_at BETWEEN '2026-05-28 16:00' AND '2026-05-28 17:00';

-- 某个变量的历史趋势
SELECT collected_at, value FROM opcua_data
WHERE var_name = '转速'
ORDER BY collected_at;
```

## 七、踩过的坑（今日新增）

5. **YAML `yes` 被解析为布尔值**: `trusted_connection: yes` → Python 读到的是 `True`，ODBC 连接串里变成了 `Trusted_Connection=True` 报错。改成 `"yes"` 加引号解决。
6. **asyncua NodeClass 比较**: `str(node_class)` 返回 `"1"` 不是 `"NodeClass.Object"`，必须用 `node_class == ua.NodeClass.Object` 直接比较枚举值。
7. **SQL Server 多实例**: 这台电脑有 4 个实例，`localhost` 默认连 2016，必须写 `localhost\MSSQLSERVER2019` 才连到 2019。
8. **Git HTTPS 连不上 GitHub**: ping 通但 TCP 443 被墙，切 SSH Key 走 22 端口解决。
