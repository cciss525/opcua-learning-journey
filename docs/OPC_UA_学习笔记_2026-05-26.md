## 一、核心概念

### 1.1 OPC UA 是什么

OPC UA 是工业设备之间通信的标准协议。简单理解：**设备把自己的数据暴露在一个"网络地址"上，任何程序都可以通过这个地址来读数据**。

类比：OPC UA 就像工厂里的"USB 接口"——管你 USB 后面接的是键盘还是鼠标，插上就能用。OPC UA 后面接的是西门子还是汇川，读数据的代码都一样。

### 1.2 关键术语

| 术语 | 解释 | 对应文件 |
|------|------|----------|
| OPC UA Server | 提供数据的服务端（运行在工厂设备或网关上） | `src/opcua_server.py` |
| OPC UA Client | 读取数据的客户端 | `src/web_dashboard.py` 里的轮询部分 |
| NodeId | 每个数据点的"门牌号"，格式 `ns=2;i=3` | 例: `ns=2;i=7` = 伺服电机转速 |
| Endpoint | OPC UA Server 的访问地址 | `opc.tcp://localhost:4840` |

### 1.3 数据流架构

```
浏览器 (http://localhost:8080)
  ↓ HTTP 请求 /api/data
web_dashboard.py (OPC UA Client + HTTP Server)
  ↓ OPC UA 协议，每 500ms 轮询一次
opcua_server.py (OPC UA Server, 模拟工厂)
  ↓ 模拟产生
虚拟设备数据 (气缸位置、电机转速、传送带速度...)
```

**两层都是"下游拉上游"，不是"上游推下游"。**

## 二、工厂现场架构

### 2.1 OPC UA Server 在工厂的四种部署位置

方案A: 工控机(最常见)         方案B: 硬件网关(无工控机时)
┌──────────┐                  ┌──────────┐
│ 工控机     │                  │ 硬件盒子   │
│ Kepware   │← S7/Modbus      │ MOXA等    │← 私有协议
│ OPC UA →  │                  │ OPC UA →  │
└─────┬─────┘                  └─────┬─────┘
      ↓                              ↓
  你的电脑 ──── OPC UA Client ────→ 连上去读就行

方案C: SCADA 自带              方案D: 你的电脑(临时调试)
┌──────────┐                  ┌──────────┐
│ 上位机     │                  │ 你的笔记本  │
│ WinCC等   │                  │ Python脚本 │← 网线直连PLC
│ OPC UA →  │                  │ OPC UA →  │

**关键**：OPC UA Server 必须跑在 7×24 在线的设备上（工控机 / 硬件协议网关 / SCADA 上位机），不能跑在你临时调试的笔记本上。你是 **OPC UA Client**（读数据的），不是 Server（提供数据的）。

## 三、工业软件概念辨析

### 3.1 工控机 vs 上位机

| | 工控机 (IPC) | 上位机 |
|---|---|---|
| 是什么 | **物理机器**，放在车间里的加固电脑 | **软件概念**，跑在上面的程序 |
| 关系 | 硬件平台，承载上位机 | 运行在工控机上的软件 |
| 举例 | 研华工控机、西门子 IPC | WinCC、组态王、自己写的 Python |
| 类比 | 一台耐操的电脑 | 这台电脑上跑的微信/Excel |

**"上位机"** 的名字来自"上位 vs 下位"——PLC 在现场，是"下位机"；监控 PLC 的软件在更上面的电脑上跑，叫"上位机"。

### 3.2 SCADA

SCADA = 上位机的"升级版"。**把全厂几百台设备的数据集中到一起，统一监控 + 分级报警 + 存储几年历史数据。**

| | 上位机（组态王级别） | SCADA（WinCC 级别） |
|---|---|---|
| 管多少设备 | 1 条线 / 几台 PLC | 整个工厂 / 整个园区 |
| 历史数据 | 通常没有 | 几年数据可回溯 |
| 报警 | 简单报警 | 分级报警 + 短信/邮件推送 |
| 价格 | 几百~几千 | 几万~几十万 |

### 3.3 软件归属关系（重要）

SCADA 是上位机的一种，不是并列概念：

```
上位机 (概念)
  ├── 组态王、MCGS、昆仑通态  ← 轻量上位机，管几条线
  └── SCADA (上位机的进化版)
        ├── WinCC          ← 西门子，整车厂标配
        ├── IFIX           ← GE，过程工业
        └── Wonderware     ← 施耐德旗下
```

**组态王是上位机的一种具体软件。WinCC 既是上位机，也是 SCADA。上位机是概念，SCADA 是上位机的子集。**

## 六、踩过的坑

1. **NodeId 不是字符串**：OPC UA Server 自动分配数字编号（`ns=2;i=3`），不是代码里写的变量名
2. **进程名不是 python.exe**：这台电脑是 `python3.13.exe`，`taskkill /IM python.exe` 杀不掉
3. **TCP 超时很慢**：连接断开后 `read_value()` 会卡几十秒，必须加 `asyncio.wait_for(..., timeout=0.3)`
4. **终端窗口关了 ≠ 进程停了**：后台进程还在跑，必须 `taskkill /F` 或 `Ctrl+C`
5. **OPC UA 自动恢复**：Client 断开后会不断重试连接，Server 重启后自动恢复（不需要重启 Client）

## 七、网络基础补充

### 7.1 IP、端口、TCP 是什么

三个概念撑起了你今天写的整个系统。先有直观理解，再看专业定义：

| 概念 | 一句话 | 类比 |
|------|--------|------|
| **IP** | 找到那台机器 | 写字楼地址: "XX路XX号" |
| **端口** | 找到机器上的哪个程序 | 房间号: "3楼302室" |
| **TCP** | 保证数据完整到达 | 顺丰快递: 拆包裹→寄走→签收，丢了重发 |

今天你的 `opc.tcp://localhost:4840` 里: `localhost` 是 IP（本机）, `4840` 是端口, `tcp` 是运输方式。

### 7.2 TCP vs UDP — 两种"快递方式"

| | TCP | UDP |
|---|---|---|
| 送达保证 | 保证，丢了自动重发 | 不保证，丢了就丢了 |
| 顺序 | 严格按序 | 后发的可能先到 |
| 速度 | 慢（要确认收货） | 快（不关心收货） |
| 用在哪 | OPC UA、HTTP/网页、文件下载 | 视频电话、直播、游戏 |
| 类比 | 顺丰快递 | 纸飞机扔进窗户 |

OPC UA 选 TCP 是因为工业数据**不能丢、不能乱**——气缸位置丢一个包你模型就卡住了。

### 7.3 在你程序里的位置

```
opcua_server.py                    web_dashboard.py
┌──────────────────┐              ┌─────────────────────────┐
│ IP: 0.0.0.0       │    TCP      │ OPC UA Client:           │
│   = 本机监听       │ ←────────→ │   连 opc.tcp://           │
│                   │   连接      │   localhost:4840         │
│ 端口: 4840         │             │       ↑        ↑          │
│   = OPC UA 服务入口│              │   TCP协议  IP:端口       │
└──────────────────┘              │                          │
                                  │ HTTP Server:             │
                                  │   监听 localhost:8080    │
                                  │   浏览器用 TCP 连它       │
                                  └─────────────────────────┘
```

### 7.4 OSI 七层（你只需要知道这个概念）

OPC UA 和 MQTT 属于**应用层**，在你熟悉的 ping/端口 之上再加一层"工业数据语法"：

```
应用层   ← OPC UA、MQTT、HTTP  （"数据怎么包装"）
            你学的是这层
传输层   ← TCP、UDP           （"怎么送达"）
            你只需要知道 IP+端口
网络层   ← IP、ping            （"设备在哪"）
            你平时就在用
```

## 八、OPC UA / HTTP / MQTT 三种协议对比

### 8.1 一句话区别

| 协议 | 设计目的 | 谁找谁 | 模型 |
|---|---|---|---|
| **HTTP** | 浏览器跟服务器要网页 | 客户端**主动请求**，服务器响应 | 请求→响应 (拉) |
| **OPC UA** | 工业设备互相通信 | 客户端**订阅**，服务器有变化就推送 | 双向读写+订阅 (推) |
| **MQTT** | 海量设备低带宽传数据 | 设备往 Broker 发，谁要听谁订 | 多对多发布/订阅 (推) |

### 8.2 直观类比

```
HTTP    = 去图书馆: "我要这本书" → 管理员给你一本
         你不问，管理员不会主动给。你问一次才响应一次。

OPC UA  = 订天气预报: "每天发给我" + "把温度改成100"
         气象局有数据就主动推，不用每天去问。你还能远程改值。

MQTT    = 微信群: 一个人发消息，群友全收到
         没人在意谁在线谁不在。发了就发了，谁爱看谁看。
```

### 8.3 在工厂里的位置

```
工厂现场                             云端/办公室
┌──────────┐   OPC UA   ┌─────────┐   MQTT   ┌─────────┐  HTTP  ┌─────────┐
│   PLC     │ ────────→ │ 工控机    │ ──────→ │ 云端平台  │ ←───  │ 浏览器    │
│ (设备数据) │ ←─────── │ Kepware  │          │ 大数据    │ ───→  │ Vue大屏  │
└──────────┘  读+写+订  └─────────┘  发布/订阅└─────────┘ 请求/响应└─────────┘
```

| 环节 | 用什么 | 为什么 |
|---|---|---|
| PLC → 工控机 | **OPC UA** | 工业级可靠，能读能写能订阅 |
| 工控机 → 云端 | **MQTT** | 带宽小，断网续传，一对多广播 |
| 浏览器 ← 服务器 | **HTTP** | 网页 Web 标准，Vue axios 底层 |

### 8.4 关键差异表

| | OPC UA | HTTP | MQTT |
|---|---|---|---|
| 实时推送 | 有（订阅） | 没有（需轮询/WebSocket） | 有（发布/订阅） |
| 数据格式 | 二进制 | 文本 JSON | 二进制 |
| 数据类型 | 内置时间戳+质量码 | 没有，手工拼 JSON | 没有 |
| 读写能力 | 能读能写 | 只能读(GET)写(POST) | 只能发(发布) |
| 适合场景 | 车间内设备通信 | Web 网页 | 跨网络、跨云端 |

**根本原因**：HTTP 是"拉"模型（你问才给），工厂里你不能每秒轮询一次"PLC 你还好吗"——带宽扛不住。OPC UA 和 MQTT 是"推"模型（有变化自己推），这才是工业场景需要的。

## 九、三种协议的安全性对比

### 9.1 基础：TLS 加密

HTTPS 里的 S 就是 TLS——在 TCP 之上加一层加密。三种协议用的锁是同一类：

```
没有加密:                    有 TLS 加密:
你 ─→ "密码123" ─→ 服务器    你 ─→ "&#xK9@!L" ─→ 服务器
      ↑ 中间人能看到                   ↑ 中间人看到乱码
```

### 9.2 HTTP → HTTPS

默认**不加密**。想安全必须手动升级：

```
HTTP → 明文裸奔
HTTPS → TLS 加密 + 证书验证服务器身份
```

没有中间状态——要么裸奔，要么全锁。验证是**单向**的（只验证服务器，不验证客户端）。

### 9.3 OPC UA（出厂自带锁）

出厂自带安全，且是**双向验证**：

```
OPC UA 安全三件套:
1. 加密传输     ← 线缆上是乱码（同 TLS）
2. 证书验证     ← 两端互相验证（PLC 验证你不是黑客，你验证 PLC 不是假冒）
3. 用户认证     ← 用户名/密码 或 证书登录
```

三种安全模式：

| 模式 | 加密 | 防篡改 | 验证身份 |
|---|---|---|---|
| None | 无 | 无 | 无 |
| Sign | 无 | 有 | 有 |
| Sign & Encrypt | 有 | 有 | 有 |

今天 opcua_server.py 跑本地测试用 None。**真实工厂必须 Sign & Encrypt。**

双向验证的原因：HTTP 怕的是数据被偷看；OPC UA 怕的是有人**远程关停设备**——PLC 要确认"这个发停机指令的人是不是授权过的"。

### 9.4 MQTT（出厂没锁，自己配）

轻量设计，安全是"自助餐"：

```
方案A: 不加锁（默认）
  MQTT → 明文，端口 1883
  跟 HTTP 裸奔一样

方案B: 加 TLS（推荐）
  MQTTS → TLS 加密，端口 8883
  需自己配置证书 → 麻烦

方案C: 用户名+密码
  在 Broker 端配置: 谁有权限发哪个 Topic
```

### 9.5 一句话总结

| 协议 | 安全性 | 理解方式 |
|---|---|---|
| HTTP | 默认无，加 S 就有 | 家门没锁，要锁就装一把 |
| OPC UA | 出厂自带锁，双向验 | 银行金库，进出都验身份 |
| MQTT | 出厂没锁，自己买 | 快递柜，想锁就设密码 |

三把锁的钥匙是同一类（TLS/证书），差别只是**默认锁死还是默认不锁**。
